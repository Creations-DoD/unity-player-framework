§ 01

High-Level System Vision

What this framework IS — and critically, what it is NOT.

◈
Core Philosophy

This is not a "Player Controller." It is a Player Entity Composition Platform — a runtime-composable infrastructure on which any game's player experience is built. The framework itself is game-agnostic. Individual games provide data, configurations, and game-specific overrides. The framework provides structure, contracts, and orchestration.

Framework Mission Statement

Design a runtime system where a fully-featured player entity can be assembled from independently developed, independently testable, independently replaceable subsystems — all orchestrated through a thin, non-prescriptive coordinator that carries no gameplay logic of its own.

Three Layers of Thinking

FRAMEWORK CORE
Interfaces & Contracts Event Bus Infrastructure Subsystem Registry Base Classes Lifecycle Management
SUBSYSTEMS
Movement Camera Weapon Combat Animation Audio Input Interaction UI Comms
GAME LAYER
Game-Specific Configs Custom Subsystem Implementations Game ScriptableObject Data Override Behaviours

What the Framework Explicitly Guarantees

  • Any subsystem can be swapped without touching any other subsystem, provided the interface contract is honoured.
  • Any subsystem can be disabled at runtime — e.g., disable MovementSubsystem during a cinematic, disable WeaponSubsystem in a safe zone.
  • Game A and Game B share the framework core and abstract subsystem contracts, but inject entirely different implementations.
  • A multiplayer game replaces IInputProvider with a NetworkInputProvider and replaces direct Transform writes with an INetworkTransform adapter — zero changes to gameplay subsystems.
  • An AI-controlled player injects an AIInputProvider — again, zero changes downstream.

Non-Goals

✕
This framework does NOT contain game logic.

Quest systems, progression, inventory, dialogue, level design, AI enemy behaviour — these all live in game-specific code that reacts to events emitted by the player framework. The framework is agnostic to what game surrounds it.

§ 02

Recommended Software Architecture

The hybrid architectural model chosen, with justification for every decision.

Primary Architecture: Mediated Component Architecture (MCA)

A deliberate synthesis of four established paradigms, chosen specifically for long-term maintainability in a multi-game, multi-developer environment:

Pattern LayerApplied ToJustification
Mediator (via PlayerEntity)All subsystem orchestrationPlayerEntity is the sole communication hub. Subsystems never call each other directly. Eliminates N×(N-1) coupling links.
Component PatternAll subsystemsSubsystems are self-contained components. Swap or extend any without touching others. Unity-native mental model for engineers.
Event-Driven ArchitectureCross-subsystem reactions, UI, AudioDecouples producers from consumers. Subsystems emit typed events; any number of listeners react with zero awareness of each other.
Interface SegregationAll subsystem contractsSystems depend on the thinnest possible contract. IMovementSubsystem can have 4 methods while the concrete class has 40.
State Machine (Hierarchical)Movement, Combat, Weapon statesEliminates boolean flag hell. Prevents invalid state combinations. Provides clean transition hooks.
Data-Driven via ScriptableObjectsAll configuration dataDesigners tune behaviour without code changes. Multiple games use the same subsystem code with different data assets.
Service Locator (scoped)Cross-cutting concerns (Audio, Debug)Avoids deep dependency chains for globally-needed services. Bounded to framework scope — not a global singleton.
Strategy PatternCamera modes, Surface detection, Weapon typesRuntime-swappable algorithms. FPS→TPS camera switch is a strategy swap, not a structural change.

Key Architectural Decisions & Rationale

Decision 1 — Subsystems Are NOT MonoBehaviours

ℹ

Subsystems are plain C# classes implementing IPlayerSubsystem. The PlayerEntity MonoBehaviour owns and drives them. This means: no GetComponent overhead, no Unity message routing, no accidental coupling through Unity lifecycle events, clean unit testability without spinning up a scene, and full control over update order. Only the root PlayerEntity and subsystems that truly require Unity callbacks (e.g. Trigger detection in InteractionSubsystem) are MonoBehaviours.

Decision 2 — Input is Fully Abstracted

The movement system, combat system, and weapon system NEVER read from Unity's Input System directly. They receive a PlayerInputFrame struct from an injected IInputProvider. This single decision makes the entire player framework playable by: human players, AI agents, network authority, and replay systems — all without modifying a single subsystem line.

Decision 3 — Three Communication Tiers

All communication between subsystems is categorised into one of three tiers, chosen by latency and coupling requirements. Mixing these tiers inappropriately is the most common source of hidden coupling. (See Section 7 for the full communication model.)

Decision 4 — Composition Over Inheritance for Weapons

A weapon is a runtime composition of modules. There is no SniperRifle : AssaultRifle : BaseWeapon hierarchy. Instead, a WeaponInstance is assembled from a WeaponDefinition ScriptableObject and a set of modules (Shooting, Reload, Recoil, ADS, Ammo). This eliminates class explosion and allows for exotic weapon combinations (e.g., a melee weapon that also shoots) with zero new classes.

§ 03

Folder Structure

Namespace-aligned, assembly-definition-separated, game-layer isolated.

⚠
Assembly Definitions Are Mandatory

Every top-level folder in the Framework requires its own .asmdef. This enforces encapsulation at compile time, prevents game-layer code from accidentally referencing framework internals, dramatically speeds up Unity's incremental compilation, and makes dependency violations visible immediately rather than at runtime.

§ 06

Dependency Map

Who is allowed to know about whom — and the rules that enforce it.

Dependency Direction Rules

✕
The Golden Rule: Dependencies only flow inward toward interfaces — never outward toward concrete implementations.

No subsystem holds a reference to another subsystem's concrete type. All references are to interfaces. All inter-subsystem communication that isn't mediated by PlayerEntity goes through the EventBus. Violating this rule even once creates the invisible load-bearing wall that makes future refactoring impossible.

DEPENDENCY RULES (read as: X depends on Y)
══════════════════════════════════════════════════════════════════

                          → PlayerEventBus (owns it)
                          → SubsystemRegistry

                          → PlayerEventBus (to emit movement events)
                          → MovementConfig (ScriptableObject)

                          → CombatConfig (ScriptableObject)

                          → Animator (Unity component, owned directly)
                          → AnimationConfig

                          → Data channel ScriptableObjects (writes to them)

AudioSubsystem       ✕→  WeaponSubsystem    (weapon audio events carry AudioClip refs)
UICommsSubsystem     ✕→  any UI class       (one-way: framework → UI, never UI → framework)
Any Subsystem        ✕→  GameLayer classes  (framework must remain game-agnostic)

Dependency Injection Strategy

The PlayerEntity assembles and injects all dependencies at composition time (Awake/Start). Subsystems receive their dependencies through constructor parameters or an Initialize(IPlayerEntity ctx) method. No subsystem uses GetComponent, FindObjectOfType, or any scene-traversal to find its dependencies.

Injected at Construction

  • IInputProvider → all input consumers
  • PlayerEventBus → all subsystems
  • ScriptableObject configs → respective subsystems
  • IMainAudioManager → PlayerAudioSubsystem

Resolved at Runtime

  • CharacterController (scene component, owned by PlayerEntity)
  • Animator (scene component, owned by PlayerEntity)
  • Camera (scene component, owned by CameraSubsystem)

Never Resolved Via

  • GetComponent<T> across subsystems
  • FindObjectOfType<T>
  • Static singleton references inside subsystems
  • String-keyed lookups

§ 08

Design Pattern Per System

One primary pattern per subsystem — chosen for fit, not fashion. With explicit justification.

SubsystemPrimary PatternSecondaryJustification
Player Entity Mediator Composite PlayerEntity knows all subsystems but subsystems don't know each other. All orchestration flows through one hub. This is not a god class — it carries zero gameplay logic. Its only job is to route, order, and own subsystems.
Input Adapter — Unity's New Input System has its own API shape. The Adapter wraps it into a PlayerInputFrame value type that the rest of the framework depends on. When Unity changes their Input API (it has happened), only the adapter changes. All downstream code is untouched.
Movement Hierarchical State Machine Strategy (surface detection) HSM prevents invalid state combinations that boolean flags cannot. You cannot be both Prone and Swimming. Cannot be Sliding and Climbing. The state machine enforces these invariants structurally, not via if/else guards scattered across the codebase.
Parkour State Machine + Strategy Template Method Each parkour action (vault, mantle, climb) follows the same Detect → Validate → Enter → Execute → Exit template, but with different geometry queries and movement curves. Template Method extracts the invariant scaffold; strategies provide the per-action variation.
Combat / Health Observer (events) Chain of Responsibility (damage pipeline) Damage should flow through a pipeline of modifiers (armor, shields, invincibility, damage type resistances) before application. Chain of Responsibility models this cleanly — each link in the chain can modify or absorb the damage. Observers react after the final value is settled.
Weapon Composition over Inheritance Factory (WeaponInstance assembly) Weapon behaviours are modules composed at runtime from a WeaponDefinition ScriptableObject. A factory (or the WeaponManager) assembles the correct module combination. Adding a new weapon variant requires zero new classes — only a new ScriptableObject with different module configs.
Weapon State State Machine (per weapon) — Each WeaponInstance has its own state machine: Idle → Firing → Reloading → Switching → Holstered. You cannot reload while already reloading. Cannot fire while switching. The state machine enforces this structurally — no guard clauses needed in fire/reload methods.
Camera Strategy (camera modes) Decorator (camera effects) FPS and TPS camera modes are interchangeable strategies implementing ICameraMode. The CameraSubsystem doesn't care which is active. Camera effects (shake, recoil, lean) are stackable decorators applied as ordered transforms to the final camera position, not mixed into mode logic.
Animation Pure Observer / Reactive — AnimationSubsystem contains ZERO gameplay logic. It only subscribes to events and player state data, then translates those into Animator parameters. This is critical: if the Animator asset is replaced (e.g. FPS → TPS rig), only the AnimationSubsystem changes. No gameplay code is touched.
Audio Observer + Object Pool Strategy (surface audio) Audio reacts to events — it never drives them. The object pool provides pooled AudioSources — never AudioSource.PlayClipAtPoint which creates/destroys GameObjects. Surface-aware footsteps use a Strategy pattern: a FootstepProfile ScriptableObject provides the correct clip set per surface type.
Interaction Visitor — The player (visitor) visits interactables. Each IInteractable accepts the visit and knows what to do. The interaction system has zero knowledge of what any specific interactable does. Adding a new interactable type requires zero changes to the framework — only a new IInteractable implementation.
UI Comms Reactive Data Binding (via SO channels) — ScriptableObject data channels are the cleanest Unity-native solution for framework→UI communication. The HUD holds a reference to the HealthDataChannel asset — not to any gameplay class. The HUD can be rebuilt from scratch with no change to the player framework.
§ 09

State Machine Design

Hierarchical State Machines — the architecture that eliminates boolean flag hell.

⚠
Flat State Machines Do Not Scale

A flat state machine with 15 movement states requires up to 15×14 = 210 transition checks. Adding state 16 requires touching every existing state. A Hierarchical State Machine (HSM) groups states logically, so a new sub-state in an existing group only requires transitions within that group. AAA movement systems require HSMs — there is no viable alternative at this scope.

Meta State Machine (Outermost Layer)

Governs the player entity's top-level behavioural mode. All other state machines only run within the Alive meta-state.

MetaStateMachine
├── ALIVE          (active during normal gameplay)
│   └── [LocomotionHSM, CombatHSM, WeaponHSM all active here]
├── DEAD           (death animation, awaiting respawn)
│   └── [All input disabled, camera enters death mode]
├── IN_VEHICLE     (player operating a vehicle)
│   └── [MovementSubsystem replaced by VehicleMovementSubsystem]
└── IN_CUTSCENE    (scripted animation, input blocked)
    └── [Camera may be under cinematic control]

Locomotion Hierarchical State Machine

LocomotionHSM
├── GROUNDED  ──────────────────────────────── (parent state — on a walkable surface)
│   ├── IDLE          (no input, velocity ≈ 0)
│   ├── WALK          (input, below sprint threshold)
│   ├── SPRINT        (sprint input + sufficient stamina)
│   ├── CROUCH        (sub-HSM: enter from Idle/Walk/Sprint)
│   │   ├── CROUCH_IDLE
│   │   └── CROUCH_WALK
│   ├── PRONE         (transition only from Crouch or Slide)
│   └── SLIDE         (sprint + crouch input on grounded surface)
│
├── AIRBORNE  ──────────────────────────────── (parent state — not on walkable surface)
│   ├── JUMPING       (initial jump phase — upward velocity)
│   ├── APEX          (near-zero vertical velocity — optional, enables coyote time)
│   └── FALLING       (negative velocity — beyond jump or fell off edge)
│
├── MANTLE    ──────────────────────────────── (animation-driven, input locked)
├── VAULT     ──────────────────────────────── (animation-driven, input locked)
├── CLIMBING  ──────────────────────────────── (wall/rope climbing)
│   ├── CLIMBING_IDLE
│   └── CLIMBING_MOVING
├── LADDER    ──────────────────────────────── (axis-restricted movement)
│   ├── LADDER_IDLE
│   └── LADDER_MOVING
└── SWIMMING  ──────────────────────────────── (buoyancy-driven physics)
    ├── SURFACE_SWIM
    └── UNDERWATER

TRANSITION RULES (selected critical ones):
─────────────────────────────────────────
GROUNDED   → AIRBORNE    : !isGrounded && !isOnLadder
AIRBORNE   → GROUNDED    : isGrounded && landingVelocity within threshold
SPRINT     → SLIDE       : crouchInput && currentSpeed > slideMinSpeed
SLIDE      → CROUCH      : slideVelocity < slideEndThreshold  (smoothly)
SLIDE      → AIRBORNE    : !isGrounded during slide
CROUCH     → PRONE       : proneInput && currentState == CROUCH_IDLE
AIRBORNE   → MANTLE      : ParkourDetector.DetectMantle() && mantleInput
ANY        → DEAD        : MetaStateMachine transition (overrides all)

Weapon State Machine (Per WeaponInstance)

WeaponStateMachine
├── HOLSTERED     (not the active weapon)
├── EQUIPPING     (animation plays, weapon cannot fire)
├── IDLE          (ready — no user input)
├── FIRING        (fire rate timer active)
│   ├── SEMI      (single shot, returns to IDLE)
│   └── AUTO      (continuous, held)
├── ADS           (aimed-down-sights, sub-states mirror IDLE/FIRING)
├── RELOADING     (locked until complete or interrupted)
│   ├── MAG_OUT
│   ├── MAG_IN
│   └── CHAMBER   (for bolt-action/pump variants)
├── SWITCHING     (weapon swap in progress)
└── NO_AMMO       (visual/audio feedback, prevents FIRING entry)

State Machine Implementation Notes

  • Each state is a class implementing IState with Enter(), Tick(float dt), Exit(). No switch statements. No enum-keyed dispatch.
  • Transitions are data — a StateTransition is a condition predicate + target state. Transitions are evaluated in priority order each tick, not embedded in state logic.
  • State enter/exit emit events automatically via the base class — AnimationSubsystem and AudioSubsystem react to these events without being called directly.
  • History states for sub-HSMs — when returning from Vault/Mantle to Grounded, the HSM remembers the last active Grounded sub-state (idle/walk/sprint) and re-enters it.
  • No GameObject.SetActive in state transitions — state machines operate on data, not on scene graph visibility.
§ 10

Development Order

The order that minimises wasted work, validates architecture early, and enables parallel team development.

◈

The ordering principle: build the load-bearing infrastructure first, then layer gameplay on top of it. Every later phase depends on the event bus, interfaces, and input abstraction being solid. Skipping Phase 1 to get visible gameplay faster is the most expensive shortcut in game development.

Phase 0 — Framework Infrastructure (Weeks 1–2)

01

Core Contracts Layer

Define all interfaces, event structs, data structs, enums, and ScriptableObject base classes. Zero game logic. Zero Unity dependency where possible. Nothing downstream can exist without these contracts. They are the shared language of the entire framework — get them right before a single implementation is written.

02

Event Bus

Typed, allocation-free publish/subscribe bus. Struct-based events only. Zero reflection. Zero string keys. Profiling hooks and per-channel debug logging from day one. All inter-subsystem communication depends on this. A slow or poorly-typed event bus becomes a production liability immediately — fix it before anyone depends on it.

03

Service Locator

Simple, scoped service locator. Provides subsystem access to shared framework services without direct coupling. Keeps subsystems decoupled from each other. Not over-engineered — resist adding features that no current use case requires.

04

Subsystem Registry + LifecycleManager

Framework backbone. Manages subsystem registration, initialisation order, and tick routing (Tick / FixedTick / LateTick). Defines the explicit tick ordering: Input → Movement → Combat → Weapon → Camera → Animation → Audio. Every downstream system depends on this contract.

05

PlayerController Skeleton

Coordinator MonoBehaviour — subsystem wiring and lifecycle routing only. No gameplay code whatsoever. Target: under 200 lines total. Sets the architectural precedent. If gameplay logic appears here in week one, it will multiply unchecked throughout the project.

06

State Machine Infrastructure

Base classes: IState<TContext>, StateTransitionTable, HierarchicalStateMachine<T>. No concrete states yet — the scaffolding only. The HSM framework must exist before any movement or weapon states are implemented. Retrofitting it onto flat if/else logic later is expensive.

07

Debug Overlay + FrameworkLogger

SubsystemMonitor window, per-subsystem log channels (independently togglable), GizmoDrawer scaffolding. Build before any gameplay system. Debug tools built in week one save ten times their cost over the project lifetime. You will need them immediately when the first state machine runs.

Phase 1 — Locomotion Foundation (Weeks 3–4)

08

Input System

Unity New Input System adapter → PlayerInputFrame pipeline. Implement HumanInputProvider and NullInputProvider. Write first unit tests. Every other subsystem consumes input. Validates the IInputProvider abstraction seam that multiplayer and AI will later exploit without touching any gameplay code.

09

Movement System — Core States

MovementHSM with Idle, Walk, Sprint, Jump, Fall super-states. CharacterController integration. The game is unplayable without this. Validates the state machine architecture with manageable scope before adding complex states like swim, climb, or parkour.

10

Ground Detection + Gravity

Physically correct ground snap, step detection, slope angle limits, and gravity application. Movement feel is decided here. Sloppy ground detection causes jitter and edge cases that compound as more states are added on top.

11

Camera System — FPS Mode

FPS camera rig. Mouse look, pitch clamp, head bob scaffold. No modifiers yet — baseline only. You need to play-test movement from the correct perspective. Camera and movement are always designed and tuned together.

12

Animation System — Base Layer

AnimatorProxy setup. Connect movement state machine events to Animator parameters. Basic locomotion blend tree for Idle/Walk/Sprint/Jump. Movement without animation feedback makes gameplay feel broken and slows feel-tuning significantly. Get the bridge working before adding weapon states.

Phase 2 — Movement Completeness (Weeks 5–6)

13

Crouch + Prone States

Extend MovementHSM. Capsule height modification, speed reduction, and transition rules into sub-states. High gameplay value, low risk. Validates that adding new leaf states to the HSM does not break existing ground states — the architecture's first integrity test.

14

Slide State

Friction-based momentum deceleration from sprint. Transitions cleanly into Crouch on slide-end. SlideConfigSO driven. Tests HSM sub-state transitions and stamina integration together in one flow.

15

StaminaModule

Stamina pool, consumption rates per state, exhaustion threshold, recovery delay — all configurable via ScriptableObject. Stamina is consumed by movement, dodge, parkour, and climbing simultaneously. Build it once here and expose it as a shared resource via events.

16

SurfaceDetectionModule

Raycast-based ground material detection mapped to SurfaceDefinitionLibrarySO. Publishes SurfaceChangedEvent on contact change. Both audio footsteps and movement friction depend on surface type. Build it once, let all consumers react via events — no direct dependencies.

17

FallDamageModule

Track peak downward velocity. On landing, emit a DamageData event into the combat pipeline when the threshold is exceeded. Validates the movement → combat event pipeline for the first time with a real, simple cross-system flow.

18

Lean State + Camera Modifier

Left/right lean from cover. LeanCameraModifier applies lateral offset to CameraSubsystem via LeanEvent — not a direct call. First real use of the camera modifier stack. Validates that modifiers are additive, independently managed, and do not interfere with each other.

Phase 3 — Weapon Foundation + IK (Weeks 7–9)

19

Weapon System — Core

WeaponManager, fixed WeaponSlot array, WeaponDefinitionSO. Weapons are data definitions until equipped — never GameObjects at rest. The largest subsystem. Establish the composition model (data + modules) now before any modules are added — changing the model later is deeply expensive.

20

FireHandler — Hitscan

Fire pipeline: check state → check ammo → apply spread → raycast → publish WeaponFiredEvent and HitDetectedEvent. No projectiles yet. Hitscan first. Validates the entire fire pipeline, damage event fan-out, and audio/VFX observer chain in the simplest possible form before adding ballistics.

21

AmmoSystem

Clip state per WeaponInstance. Global reserve pools per calibre. Tactical vs. empty reload distinction in AmmoSystem contract. Ammo logic is referenced by fire, reload, UI, and save systems simultaneously. Define it cleanly once rather than patching it across four places.

22

ReloadHandler

Tactical and empty reload paths with per-weapon interruptibility rules defined in WeaponDefinitionSO. Reload is the most animation-coupled weapon flow. Getting the event → animation → complete → refill chain right here prevents fragile timing hacks later.

23

RecoilSystem + Camera Modifier

Per-weapon RecoilProfileSO mathematical model. Publishes RecoilEvent — Camera and AnimationSystem consume it independently. Validates that two independent subsystems (camera visual recoil and animation layer recoil) react to the same event with zero coupling between them.

24

AdsSystem + FOV Modifier

ADS state management, zoom level, sensitivity reduction. FOVModifier driven by AdsStateChangedEvent — not a direct camera call. First compound modifier: ADS affects FOV, spread, weapon sway, and camera simultaneously, all via events. Validates the multi-consumer event model.

25

WeaponAudioHandler + WeaponVFXHandler

Both are pure event consumers: subscribe to WeaponFiredEvent, ReloadedEvent, EmptyEvent. Never called directly by any other system. Validates the observer pattern at scale. Five or more subsystems react to WeaponFiredEvent with zero coupling between any two of them.

Phase 4 — Combat + Polish (Weeks 10–12)

26

IK System + Procedural Animation

Foot IK on uneven terrain. Hand IK on weapon grip. Aim IK for full-body spine-to-camera tracking. ProceduralAnimationSystem: weapon sway, breathing offset layer. Moved to Phase 3 for full-body FPS. The full skeleton rig requires IK from the moment the player can see their own body — foot placement, hand grips, and camera-driven spine rotation cannot be retrofitted cleanly after the animation system is in use.

27

HealthSystem + DamageProcessor

Damage pipeline: armor → resistance → invincibility frames → apply HP deduction. Publishes HealthChangedEvent and DamageReceivedEvent on completion. Validates the Chain of Responsibility pattern for damage and the cross-entity event model used by every combat system from here forward.

28

HitDetectionSystem

Hitscan and projectile-collision variants per weapon config. Publishes typed HitDetectedEvent into the damage pipeline. Separating hit detection from damage application is critical for multiplayer authority design. Do not merge them.

29

HitReactionSystem + DeathSystem

Hit reactions as discrete states. DeathModule owns the full death state transition — death is a state, never a boolean flag. The death lifecycle fans out to camera, animation, audio, UI, and game layer simultaneously. Validates all event fan-out paths under one flow.

30

MeleeSystem + ComboSystem

Melee hitbox activation, combo timing windows, finisher detection. Combo chain modelled as a flat state machine with configurable timing thresholds. Tests time-window-based state machine design — a different challenge from movement or weapon states and good coverage of the transition system.

31

PlayerAudioSystem

FootstepModule (surface-aware), BreathingModule (stamina-driven), CombatVoiceModule (health events). All purely event-driven. Audio built in isolation against placeholder events is always wrong. Build it now against real event data flowing from real gameplay systems.

32

Interaction System + UIBridge

IInteractable priority queue, overlap detection, interaction prompt events. UIBridge translates gameplay events to ScriptableObject data channels for HUD. Completes the feedback loop: a player can now interact, see health, ammo, and stamina updates — all without any UI class touching gameplay code.

Phase 5 — Advanced Systems (Weeks 13–15)

33

Parkour System

ParkourDetector geometry queries (raycasts + shape casts). Vault, Mantle, Climb as Command-pattern sequences. Separate subsystem from Movement — independently disableable. Parkour geometry detection is complex. Build after core movement is stable so the geometry queries can be validated against real level data.

34

ProjectileSystem

Pooled projectile lifecycle. Burst-compiled IJobParallelFor + RaycastCommand for multi-projectile weapons. Flyweight definition via WeaponDefinitionSO. Projectiles are the highest-frequency short-lived objects in the framework. Pool-first from the start — retrofitting pooling onto physics projectiles is painful.

35

Swimming + Ladder States

WATER super-state with SwimSurface, SwimUnderwater, Wade leaf states. Water volume detection, buoyancy, breath metre. VERTICAL Ladder state with axis-restricted movement. Validates HSM super-state extensibility. New super-states are added without modifying any existing state class — this is the open/closed test.

36

Dodge State

Directional dodge/roll. Invulnerability frames. Stamina burst cost. Housed in EVASIVE super-state. Tests the stamina burst cost pattern and invulnerability frame timing used heavily in action game mechanics — important to get right before content production.

37

TPS Camera Mode

TPSCameraMode strategy swap. Orbit camera with over-shoulder collision avoidance. Full body visible. Defined FPS→TPS switch protocol. FPS must be solid first. TPS uses the same modifier stack — validates that the Strategy pattern swap changes nothing else in the camera system.

Phase 6 — Production Readiness (Weeks 16–18)

38

Save/Load State Hooks

Identify all authoritative runtime state. Tag with [AuthoritativeState] attribute. Expose a serialisable snapshot interface on every owning subsystem. Retrofitting save/load after release is one of the most painful engineering tasks. The snapshot hooks must exist now before game-layer progression depends on them.

39

Editor Tools

StateMachineVisualizer (live graph in Editor), WeaponConfigEditor (custom SO inspector), SubsystemMonitor (runtime diagnostics window). Editor tools built during development save their cost every day they exist. Ship them before content production begins — designers depend on them immediately.

40

Full Test Suite

EditMode: state machine logic, event bus correctness, damage pipeline. PlayMode: movement flow, weapon fire chain, combat lifecycle. Framework code without tests is framework code you cannot refactor safely. Tests are the licence to refactor. Write them now while the intent is clear.

41

Optimisation Pass

GC allocation audit under Unity Memory Profiler. Object pool coverage verification. Physics raycast batching check. Animator hash cache validation. Performance issues discovered in week 16 are almost always architectural. Fix them before game-specific content makes them load-bearing and immovable.

42

Multiplayer Boundary Audit

Review all [AuthoritativeState] tags. Document every client-predicted path. Define the NetworkInputProvider implementation contract for the networking team. A boundary audit costs 3–5 engineering days now. Discovering missing authority boundaries under a multiplayer deadline costs months.

43

API Documentation

XML docs on all public interface methods. Architecture guide. Subsystem ownership register. Usage examples for all common extension patterns. Documentation written by the people who built the system is the only accurate documentation. Write it now while the decisions and their reasons are still fresh.

§ 11

Scalability Planning

How the framework accommodates growth without requiring architectural changes.

Adding a New Game

  • Create a new Game.asmdef under Games/[GameName]
  • Create ScriptableObject config assets for all subsystems
  • Extend PlayerEntityBase if game-specific logic needed
  • Inject game-specific IInputProvider variant if needed
  • Zero changes to framework code

Adding a New Weapon Type

  • Create a new WeaponDefinition ScriptableObject
  • Configure module settings in the asset
  • Optional: create a custom module variant implementing IWeaponModule
  • Zero new weapon classes required
  • Zero changes to WeaponSubsystem

FPS → TPS Switch (Runtime)

  • Call cameraSubsystem.SetMode(CameraMode.TPS)
  • AnimationSubsystem listens to mode change event, swaps animation rig
  • WeaponSubsystem adjusts aim calculation for TPS offset
  • Zero structural changes — pure strategy swap

Multiplayer Integration

  • Replace IInputProvider with NetworkInputProvider
  • Wrap Transform writes with INetworkTransform adapter
  • Route DamageEvents through server validation layer
  • All subsystems remain unchanged
  • All gameplay logic remains unchanged

Adding a New Subsystem

  • Define a new interface (e.g., IVehicleSubsystem)
  • Implement it as a class extending PlayerSubsystemBase
  • Register it in PlayerEntity's composition
  • Existing subsystems unaffected

Adding Movement States

  • Create a new state class implementing IState
  • Register it with the appropriate HSM layer
  • Define transitions in/out of the new state
  • Existing states and transitions unaffected

Versioning & Breaking Change Strategy

  • Interface versions — When an interface must change in a breaking way, create IMovementSubsystem_V2 and migrate at controlled cadence. Never silently change an existing interface contract.
  • Event struct versioning — Add new fields with defaults. Remove fields only in major framework versions with migration guides. Do not rename event structs.
  • ScriptableObject config versioning — Use Unity's serialization migration hooks. Keep configs additive. Maintain backward compatibility for at least one major version.
  • Assembly definition isolation — Breaking changes within a subsystem are invisible to other subsystems, because they depend on the interface, not the implementation. This is the primary scalability mechanism.
§ 12

Performance Risks

Identified risks, severity, and mitigation strategies. Architectural decisions have performance consequences.

HIGH RISK

Event Bus Allocation in Hot Paths

If event structs are heap-allocated (classes not structs), or if the event bus uses boxing/unboxing, high-frequency events (per-frame movement updates, per-shot weapon events) will generate garbage and trigger the GC.

✓ Mitigation: All event types are value types (structs). Event bus uses generic delegates with struct constraints. Validate with Unity's Memory Profiler before shipping. Reserve EventBus for state transitions and meaningful events — NOT per-frame continuous data.

HIGH RISK

Physics Query Volume — Surface Detection & Interaction

SurfaceDetectionModule and InteractionDetector both perform raycasts or OverlapSphere queries. If these run every frame naively, a game with 32 networked players performing surface queries generates thousands of physics calls per frame.

✓ Mitigation: Surface detection runs only when velocity or surface contact changes. InteractionDetector uses OverlapSphereNonAlloc with a pre-allocated buffer. Physics queries are rate-limited to a configurable interval (e.g., every 3 frames for surface type — surface type rarely changes at 60fps).

MEDIUM RISK

State Machine Transition Overhead

If state transition conditions are evaluated every frame for all states, and if those conditions involve physics queries or expensive computations, the state machine itself becomes a performance cost.

✓ Mitigation: Transition conditions must be O(1) — pure data comparisons, no queries. Physics results are cached by the appropriate module and the state machine reads the cache. Transition evaluation is ordered by most-likely-to-fire first (early exit).

MEDIUM RISK

Animator Parameter Writes

Setting Animator parameters by name (string hash lookup) every frame for many parameters is measurably expensive. With complex blend trees and IK layers, this can accumulate.

✓ Mitigation: Pre-hash all Animator parameter names at initialization using Animator.StringToHash(). Cache hashes as static int constants. Only write parameters when values change — dirty-flag pattern on all Animator parameter writes in AnimationSubsystem.

MEDIUM RISK

AudioSource Management

Creating/destroying AudioSources for footsteps, weapons, and voice generates garbage and spikes. AudioSource.PlayClipAtPoint is a known offender — it creates a new GameObject.

✓ Mitigation: AudioPool maintains a pool of pre-warmed AudioSource components. All audio playback goes through the pool. AudioPool size is tuned per scene type. Never call PlayClipAtPoint anywhere in the framework.

MEDIUM RISK

LINQ and Delegate List Allocation

C# closures, LINQ queries, and delegate list iteration in hot paths (Tick methods) generate garbage. This is particularly dangerous in event bus subscriber management.

✓ Mitigation: No LINQ in any Tick() method. Event bus subscriber lists are pre-allocated arrays, not List<T>. Delegate invocation uses cached delegate references. Profiler rule: zero allocations in any method called every frame with more than one player in scene.

LOW RISK

ScriptableObject Data Channel Write Frequency

UICommsSubsystem writes to SO data channels on events. If events are high-frequency (e.g., continuous stamina drain), UI updates may be more frequent than necessary.

✓ Mitigation: Data channels implement change-threshold filtering — only notify if value changed by more than a configurable epsilon. UI elements should also implement dirty-checking before redrawing.

§ 13

Architecture Mistakes To Avoid

The specific failure modes that will make this framework unmaintainable in 12 months. Not theoretical — each is a real trap seen in production.

✕
Mistake 1 — PlayerController as a God Object

Any script that starts accumulating fields like isGrounded, isSprinting, currentWeapon, health, animator, audioSource, and camera in the same class is a god object forming. The player controller should contain ONLY subsystem references and lifecycle orchestration. If you find yourself writing gameplay logic in PlayerEntity, it belongs in a subsystem.

✕
Mistake 2 — Boolean Flag State Management

Replacing a proper state machine with isGrounded, isJumping, isCrouching, isSprinting, isSliding boolean flags. These are not a state machine. They allow structurally invalid combinations (isJumping && isSwimming = true simultaneously). They require O(n²) guard clauses. They break instantly when new states are added. Use the HSM.

✕
Mistake 3 — Direct Input Reading in Subsystems

Any subsystem calling Input.GetAxis(), Keyboard.current.spaceKey, or playerInput.actions["Jump"] directly. This permanently couples the subsystem to human input, making AI control, replay, and multiplayer require hacking the subsystem internals. All input flows through IInputProvider → PlayerInputFrame, period.

✕
Mistake 4 — String-Based Events or Messages

Using SendMessage("OnDamaged"), BroadcastMessage(), or a Dictionary<string, Action> event bus. These have zero compile-time type safety, break silently when method names change, perform reflection at runtime, and are impossible to trace in a debugger. Use typed struct events exclusively.

✕
Mistake 5 — Subsystems Referencing Each Other Directly

Writing this.weaponSubsystem = GetComponent<WeaponSubsystem>() inside MovementSubsystem. Or player.CameraSubsystem.ShakeCamera() from inside CombatSubsystem. These create hidden coupling links that make it impossible to replace or test any subsystem in isolation. All cross-subsystem communication goes through the event bus or through PlayerEntity as the mediator.

✕
Mistake 6 — Gameplay Logic in AnimationSubsystem

Writing if (currentAmmo == 0) TriggerEmptyAnimation() in the AnimationSubsystem, or making the AnimationSubsystem call into WeaponSubsystem to check state. AnimationSubsystem is a pure reactor. It contains zero gameplay conditionals. It only converts events and state data into Animator parameters. Any gameplay logic found here is in the wrong place.

✕
Mistake 7 — Inheritance Hierarchy for Weapons

Building a weapon class hierarchy: BaseWeapon → RangedWeapon → AssaultRifle → M4A1. This is the classic way to paint yourself into a corner. What is the M4A1 with a grenade launcher attached? Where does a knife that can also throw go? Composition over inheritance. Weapons are WeaponInstances assembled from module configurations.

✕
Mistake 8 — Camera Logic Embedded in Movement

Putting camera rotation, FOV changes, or camera shake inside the MovementSubsystem. Camera and movement are two separate concerns that happen to share player position data. A TPS camera implementation does not require refactoring movement. They must remain separate from the beginning.

⚠
Mistake 9 — Over-Engineering the Event Bus

Making the event bus support priority ordering, event queuing, cancellation, async dispatch, and network serialization all at once. Start with a simple, fast, synchronous typed event bus. The complexity budget for the event bus should be extremely low — its value is simplicity and type safety, not features. Add capabilities only when a specific requirement demands it.

⚠
Mistake 10 — Skipping Assembly Definitions

Without .asmdef files, the dependency rules you've designed exist only as a gentleman's agreement in documentation. Within 6 months of production, someone will add a reference "just for now" that violates the dependency map. Assembly definitions enforce the architecture at compile time. They are not optional for a multi-game framework.

§ 14

AAA Best Practices

Engineering standards that separate professional production code from prototype code.

Code Quality Standards

  • Every subsystem is independently unit-testable — No subsystem requires a running Unity scene to test its core logic. Input, state machines, damage calculations, weapon fire rate — all testable with mock dependencies in EditMode tests.
  • Zero allocations in hot paths — Every method called in Tick(), FixedTick(), or LateUpdate() must be profiled for zero GC allocation. This is a shipping requirement, not a recommendation.
  • Interface-first design — Define the interface before the implementation. The interface is the contract. If you can't define the interface cleanly, the responsibility boundary is wrong.
  • Explicit update order — PlayerEntity drives the explicit Tick() order: Input → Movement → Weapon → Combat → Camera → Animation → Audio. Never rely on Unity's implicit script execution order for gameplay-critical sequences.
  • No magic numbers — Every configurable value lives in a ScriptableObject config asset or a constant with a named identifier. No hardcoded values like if (health < 20f) in code.

Debug & Tooling Standards

  • Runtime debug visualisation — Every subsystem provides Gizmo/Handle drawing in the editor. Surface detection raycasts visible. Interaction detection zones visible. State machine current state visible in Inspector. This is not optional — it is how you debug and tune gameplay.
  • Subsystem monitor — A runtime debug window showing: current state per state machine, event bus message throughput, subsystem enable/disable status, performance budget per subsystem. Built as part of Phase 1.
  • Event bus debug logging — Every event emission should be optionally loggable with a toggle. Invaluable for debugging state transition issues without a debugger attached.
  • Subsystem hot-reload testing — Design subsystem enable/disable to be triggerable from the editor at runtime. This allows rapid iteration on disable edge cases without game restarts.

Data Standards

  • ScriptableObject configs are the source of truth — Not hardcoded constants in MonoBehaviours. Not values in prefabs. The config asset is what a designer edits. The code reads from it.
  • Weapon definitions are pure data — A designer can create a new weapon by duplicating and modifying a WeaponDefinition asset. No programmer involvement required for weapon feel iteration.
  • Platform-variant configs — Design the config system to support platform overrides. Console vs PC may need different sensitivity defaults, different control scheme defaults. The config architecture should support this from day one.

Team Scalability Standards

  • Each subsystem is owned by one engineer — Clear ownership prevents both orphaned code and contradictory changes. The subsystem interface is the contract between engineers.
  • No subsystem merge conflicts cross-contaminate — Because subsystems are in separate files (and ideally separate directories), two engineers working on MovementSubsystem and WeaponSubsystem simultaneously never have merge conflicts in each other's files.
  • New team members can contribute in their subsystem without understanding the whole framework — The subsystem interface is the complete specification for what a new subsystem engineer needs to understand. This is the scalability measure of good architecture.
§ 15

Professional Recommendations

Senior-level strategic guidance on how to execute this architecture successfully over a multi-year production timeline.

Before Writing a Single Line of Gameplay Code

✓
Write the interfaces first. All of them. Review them with your team.

Spend a full sprint writing only interfaces, event structs, and the PlayerInputFrame struct. Have every engineer on the team review the interfaces before any implementation begins. The interfaces ARE the architecture. Implementation mistakes are recoverable. Interface design mistakes compound for years.

The Seams That Matter Most

Of all the abstraction seams in this architecture, three are load-bearing for future scalability:

  1. IInputProvider — This single interface is what separates a singleplayer-only framework from a multiplayer-ready one. Do not let anything breach this seam.
  2. The Event Bus contract — All events as value types, all subscriptions type-safe, zero reflection. Every shortcut taken here becomes a performance or maintainability problem at scale.
  3. Assembly Definition boundaries — The only compile-time enforcement of your dependency rules. Without them, dependency rules exist only in documentation that nobody reads in crunch.

What to Build Lean vs What to Over-Engineer

SystemRecommendationReason
Event Bus Keep it lean — synchronous, typed, no features Complexity here is a liability. Simplicity is the feature.
State Machines Invest here — full HSM from the start Retrofitting HSM onto a flat state machine costs more than building it right initially.
Weapon Modules Invest here — composition from day one Weapon variety is the content scalability test. Inheritance-based weapons cannot support the variety that production demands.
Audio System Build early, invest in the pool Audio systems refactored late in production break dozens of things. Build the pool and surface detection correctly in Phase 3, not as post-launch patch.
Debug tooling Build it in Phase 1 Debug tools that exist during development save 10× their build cost in debugging time. Build the SubsystemMonitor before any gameplay code.
Multiplayer integration Design seams now, implement later IInputProvider and INetworkAdapter need to exist in the architecture from day one. The implementations can wait. Adding seams after the fact requires touching every subsystem.
ScriptableObject configs Design thoughtfully, not exhaustively Don't try to expose every value on day one. Start with what designers need to tune. Add fields incrementally. Changing SO structure mid-production is manageable if done carefully.

Long-Term Maintenance Philosophy

  • Architecture rot is gradual — The first "just this once" violation of the dependency rules is the one to catch and correct. Every subsequent violation becomes easier to justify. The lead engineer must enforce boundary discipline actively, not just document it.
  • Interfaces evolve — manage it deliberately — Every interface change should go through a brief impact review. Document why the interface changed. This creates a record of architectural decisions that new team members can read to understand why the system is shaped the way it is.
  • The framework should feel boring to use — Good infrastructure is invisible. If engineers are frequently fighting the framework, wrestling with its conventions, or inventing workarounds, the architecture has drifted from its users' needs and needs revisiting.
  • Measure the right things — Test coverage on subsystem unit tests, GC allocation per frame in hot paths, and time-to-integrate a new subsystem are the metrics that tell you if the architecture is healthy. Counting lines of code or number of classes tells you nothing useful.

Final Architect's Note

The hardest thing about this architecture is not building it — it is maintaining the discipline to keep it clean under production pressure. Build the structural enforcement in Phase 1. It will pay back its cost every single week of production that follows.
